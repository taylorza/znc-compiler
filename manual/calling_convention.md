# Calling Convention

This page documents the ZNC calling convention, how arguments are laid out on the stack, how the compiler references arguments and locals via the `ix` frame, and the meaning of `__znccall(n)` annotations.

## Overview

- Arguments are passed on the stack, left-to-right (the first parameter is pushed first, then the second, etc.).
- By default the caller is responsible for cleaning the arguments from the stack (caller‑cleanup). This is the default behavior (`__znccall(0)` or no annotation).
- `__znccall(1)` requests callee‑cleanup (callee pops/adjusts the stack before returning). `__znccall(1)` is NOT permitted for variadic functions.
- Return values are returned in `HL` for non-void scalar/word results (bytes/char are returned in `L` with sign/zero‑extension where appropriate).

## Typical function prologue

Assembler and compiler-generated functions use the following prologue (non-top-level functions):

```asm
  push ix
  ld ix,0
  add ix,sp
```

After that sequence `ix` is a stable frame pointer that can be used to reference the saved `ix`, the return address, all arguments, and local storage.

## Stack layout (bytes) relative to `ix`

With the prologue above the stack looks like this (offsets are bytes from `ix`):

```
  ix+0 : saved IX (low)
  ix+1 : saved IX (high)
  ix+2 : return address (low)
  ix+3 : return address (high)
  ix+4 : last argument (argument N) low byte
  ix+5 : last argument (argument N) high byte
  ix+6 : argument N-1 low byte
  ix+7 : argument N-1 high byte
  ...
  ix+(4 + 2*(N-1)) : first argument (argument 1) low byte
  ix+(5 + 2*(N-1)) : first argument (argument 1) high byte
```

- With `N = func_arg_count` the k-th argument (1‑based) low byte is at `ix + 4 + 2*(N - k)` and the high byte at `ix + 5 + 2*(N - k)`.
- Equivalently, using the compiler's zero‑based `arg_index` the helper computes the base offset as `base_offset = 2 + (func_arg_count - arg_index) * 2`, which points at the low byte for that parameter.
- The assembler shorthand `ld l,(ix+K)` works when `K` fits in signed 8‑bit range (-128..127). If the offset is outside that range the compiler emits code to compute an absolute address in `HL` and access memory via `HL`.

### Variadic functions

- For variadic functions the compiler pushes an extra 2‑byte word containing the count of variadic arguments immediately after the fixed arguments (so the count occupies the next argument slot). Call generation pushes fixed args left‑to‑right, then pushes the variadic count, then executes the call.
- Because the count is pushed, the runtime helpers and `va_start`/`va_arg` support use that layout to locate the variable arguments. The compiler emits helper RTL sequences to compute variadic offsets when needed.
- Note: `__znccall(1)` (callee cleanup) is disallowed for variadic functions — the compiler enforces this.

## Compiler view / symbol offsets

The compiler assigns symbol offsets and uses these formulas to compute IX-relative addresses:

- Arguments: the compiler computes the low‑byte offset for an argument symbol using

  `base_offset = 2 + (func_arg_count - arg_index) * 2`

  where `arg_index` is the argument's position in the declaration (zero‑based in the compiler internal representation) and `func_arg_count` is the total number of (declared) arguments. The returned `base_offset` points at the low byte for that parameter; the high byte is `base_offset + 1`.

  (At the assembly level this corresponds to the mapping above: last argument (argument N) low = `ix+4`, argument N-1 low = `ix+6`, ..., first argument (argument 1) low = `ix+(4 + 2*(N-1))`.)

- Locals: local variables are allocated below the stack pointer and are addressed with negative IX offsets. The compiler returns a base offset pointing at the low byte computed as:

  `base_offset = - (bp_offset + size)`

  where `bp_offset` is the compiler's byte offset for the local (increasing as locals are declared) and `size` is the storage size of the symbol. In assembler you will see locals referenced as `(ix-<n>)` or `(ix+<negative>)`.

The compiler emits a helper path when an IX offset does not fit an 8‑bit signed displacement: it computes the full address in `HL` (copying `IX` into `HL` then adding the 16‑bit offset) and reads/writes through `HL`.

## `__znccall(n)` semantics

- `__znccall(0)` (default) — caller cleans the stack after the call. The compiler generates the call and then removes argument words from the stack (using `pop` sequences or by adjusting `SP`).
- `__znccall(1)` — callee cleans the stack. The function epilogue generated by the compiler will remove the argument words before returning (the compiler implementation pops the return address into `BC`, adjusts `SP` by `arg_count * 2`, then pushes the return address back and returns). This is equivalent to a `stdcall`‑style callee cleanup. Example:

```c
int add(int a, int b) __znccall(1) {
  return a + b;
}
```

- Important: Do not use `__znccall(1)` for variadic functions — the compiler will report an error.
- If you write an assembler function (`__asm__`) and use `__znccall(1)`, be sure your asm epilogue performs the same cleanup (adjust `SP` to remove arguments) before returning.

### Optimized assembler functions with `__znccall(1)`

When implementing a function entirely in assembler and using the callee‑cleanup convention (`__znccall(1)`), you can skip the normal frame prologue (`push ix` / `ld ix,0; add ix,sp`) and instead pop the return address and argument words directly from the stack into registers. This reduces code size and avoids allocating an IX frame.

Key points:

- On entry (immediately after `call`) the stack contains the return address at `SP`/`SP+1`, and the pushed arguments below that (the last pushed argument is nearest the return address).
- Typical pattern:

```asm
  pop bc    ; BC = return address (low->C, high->B)
  pop hl    ; HL = last argument (argument N)
  pop de    ; DE = argument N-1
  ; ... pop more args as needed (in reverse order)
  ; perform computation, place return value in HL
  push bc   ; restore return address
  ret
```

- Always restore the return address before returning (push/pop around `ret`). Failing to do so will jump to an incorrect address and crash.
- Because no IX frame is set up you cannot use `ix`‑relative addressing for arguments or locals. If you need local storage you must allocate it manually on the stack.
- This optimization is invalid for variadic functions (the compiler disallows `__znccall(1)` for variadic functions) and must not be used when `va_start`/`va_arg` helpers are required.
- Preserve or save/restore any registers the caller expects to be preserved; the example below is minimal and clobbers `DE`/`HL` as usual for return values.

Example — a 16‑bit add implemented in asm with callee cleanup:

```c
int add(int a, int b) __znccall(1) __asm__ {
_
  pop bc     ; BC = return address
  pop hl     ; HL = b (last pushed)
  pop de     ; DE = a (first pushed)
  push bc    ; restore return address

  add hl,de  ; HL = a + b
  
  ret
}
```


## Examples

- Reading the second argument (word) in inline assembler:

```asm
  push ix
  ld ix,0
  add ix,sp
  ld l,(ix+6)  ; arg2 low
  ld h,(ix+7)  ; arg2 high
  ; HL now = arg2
  pop ix
```

- Writing a local 16‑bit value at a 2‑byte local reserved at compiler offset `bp`:

```asm
  ; base = -(bp + 2)  ; compiler-calculated value
  ld l, <value low>
  ld h, <value high>
  ld (ix-<n>),l
  ld (ix-(<n>-1)),h
```

## Tips & Caveats

- Always ensure both caller and callee agree on `__znccall` for direct calls, delegates (function pointers), and assignments to delegate‑typed variables. A mismatch will corrupt the stack.
- Use `(ix+offset)` only when `offset` fits signed 8‑bit; the compiler will generate a safe HL‑based addressing sequence otherwise.
- When mixing C and inline assembly, follow the prologue/epilogue conventions above (or call into compiler‑generated wrappers) so stack/IX layout remains consistent.
